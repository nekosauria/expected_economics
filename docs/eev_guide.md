# 需求評估：這個功能到底該不該做？

> 本文件的評估方法，建立於 **期望值（Expected Economic Value, EEV）** 的數學基礎之上。
> EEV 公式為：**EV = Σ (Pi × Vi) - C**
> 即每個結果的「發生機率 × 對應價值」加總後，扣除開發成本。
> 用白話說：**不是看功能酷不酷，而是看期望報酬能不能打贏開發成本。**

---

在軟體開發中，我們常遇到：「這個功能看起來很酷，但做起來很耗時」。為了避免資源浪費，請在開發前與需求方（PM / 客戶）一起走完以下三個步驟。

---

## 第一步：釐清需求本質（直覺四象限）

不要先想技術細節，先問自己：**這個功能的價值，對得起它的開發成本嗎？**

| 價值 vs 成本 | 低成本 | 高成本 |
| :--- | :--- | :--- |
| **高價值** | 🥇 **必做**（快做！） | 🥈 **規劃做**（核心功能） |
| **低價值** | 🥉 **閒暇做**（有空再說） | ❌ **別做**（浪費資源） |

- **高價值：** 能幫公司賺錢、幫客戶省下大量時間、或解決用戶的核心痛點。
- **高成本：** 需要大量人力、數週開發時間、或未來維護難度極高。

這個象限對應 EEV 中的 **Vi（價值）** 與 **C（成本）** 兩個軸。當 Vi 遠大於 C，才是值得投入的需求。

---

## 第二步：防禦性提問（拆解必要性）

當有人提出一個「看起來很酷、成本很高」的需求時，請不要直接答應，先問這三個關鍵問題：

**1. 「不做會怎樣？」**

這個問題在評估 **Pi（發生機率）** 與潛在損失。
- 如果答案是「沒差，只是看起來比較帥」→ 低價值需求，EEV 很可能為負。
- 如果答案是「客戶會流失」、「系統會崩潰」→ 不做的損失極高，屬於高優先需求。

**2. 「到底有多少人會用？」**

受益人數直接影響 **Vi 的期望乘數**。如果只有 1% 的用戶需要，這個功能的期望價值就只有全體上線的 1/100。
考慮用更輕量的解法，而不是大改底層架構。

**3. 「能不能先做簡化版（MVP）？」**

MVP 的本質是**降低 C（成本）**，同時透過真實數據提高對 Pi 的估算準確度。
先花 2 天驗證假設，再決定要不要投入完整開發，是 EEV 框架下最有效的止損策略。

---

## 第三步：優先級打分（避免爭論）

如果大家意見分歧，請用數字說話。請需求方與工程師共同為功能評分（1～5 分）：

| 維度 | 說明 | 對應 EEV 元素 |
| :--- | :--- | :--- |
| **影響力** | 有多少用戶受惠？（1 = 極少，5 = 全部） | Vi 的乘數 |
| **急迫性** | 不做的話，業務現在會停擺嗎？（1 = 隨時可做，5 = 現在就要） | Pi 的權重 |
| **複雜度** | 開發起來多困難？（由工程師打分，1 = 簡單，5 = 很難） | C 的代理指標 |

**優先分數公式：**
優先分數 = (影響力 + 急迫性) ÷ 複雜度

| 分數區間 | 建議行動 |
| :--- | :--- |
| > 2.5 | 優先級極高，排入下個 Sprint |
| 1 ~ 2.5 | 中等優先，列入 Backlog 持續觀察 |
| < 1 | 又難又不急，暫緩或放棄 |

---

## 給需求者的建議

工程師不是想拒絕你的需求，而是想確保資源花在對的地方。**EEV 框架的核心精神是：讓每一分開發資源，都能帶來可量化的期望回報。**

以後提需求，請準備好這三句話，我們就能快速達成共識：

1. 這個功能能幫公司創造什麼具體價值？
2. 除了這個做法，有沒有更簡單的替代方案？
3. 我們如何測量這個功能上線後是真的有效的？

---

*本文件基於 Expected Economic Value（EEV）框架，旨在減少溝通誤解，讓團隊資源集中於真正具備經濟效益的開發任務。*
